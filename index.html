The Problem: Three Failure Modes

Interns + Claude Code without structure → all three hit simultaneously

The Chaos Problem

No framework → no consistency. Prompting wildly, reinventing the wheel, code ranging from brilliant to broken with no way to tell which is which.

The Visibility Problem

Managers have no idea what's happening. Wrong direction at day 3 gets caught at week 6 — if at all.

The Knowledge Drain

When interns leave, everything leaves with them. Research, patterns, dead ends, context. Next cohort starts at zero.

AI tools amplify human behaviour — without structure, they amplify chaos just as readily as they amplify capability.

The ATLAS Framework

Four interlocking components — each one solves a failure mode

01 — Knowledge Vault

Shared Obsidian repo, Git-backed. Per-person folders + shared research layer. Solves: knowledge drain.

02 — CRISPY Method

Structured problem-solving before touching code. Clarify → Research → Identify → Structure → Produce → Yield. Solves: chaos.

03 — Oversight Layer

Git as ground truth. Daily pushes, weekly review, PR as learning moment. Solves: visibility.

04 — Skill Development

AI foundations week 1. Prompting standards. AI hygiene. Knowledge contribution social contract. Solves: all three.

Why Obsidian?

  • → Wikilinks make cross-referencing effortless
  • → Git-backed — full history, diffs, blame
  • → Graph view reveals knowledge clusters
  • → Markdown = LLM-friendly (feed notes back to Claude)
  • → Local-first, company-owned
Interns can see each other's folders — cross-pollination happens naturally. Learning from adjacent sessions, not just lectures.

The Session Log — As a Reflection

Claude already knows what happened. The daybook is about the intern's perspective on it.

How it works

1 Intern runs /daybook at end of session
2 Claude leads: "Here's what I saw us work on today…" — intern confirms, adds context, flags anything missed (including work done outside Claude)
3 Probes deeper: dead ends, what surprised you, how are you feeling about the task
4 Writes structured log to vault automatically — nothing to format
⚡ Session-start hook nudges interns if yesterday's log is missing — fires once per day, not on every message

Why the flip matters

Claude already knows. Asking "what did you do?" wastes the context. Leading with a summary means the intern reacts — much lower friction.

Captures the full picture. The intern adds what Claude didn't see — meetings, offline research, manual testing.

Same output — structured log in the vault — but the intern actually wants to do it.

Component 2: CRISPY Methodology

Structured problem-solving before a single line of code

PhaseWhat happens
C — ClarifyState the problem. Constraints. What does success look like?
R — ResearchInvestigate the codebase, existing patterns, relevant docs — before building
I — IdentifyList approaches. Pick one. State why.
S — StructureBreak implementation into testable vertical slices. Identify risks.
P — ProduceBuild it. Follow the plan.
Y — YieldReview. Does it match success criteria? What was learned?

Without CRISPY

Prompt → paste → hope → wonder why it broke

With CRISPY

Understand → research → plan → build → verify

Component 4: Skill Development

Week 1 before touching any code

AI Foundations

How LLMs work (enough to avoid blind trust). Claude Code vs ChatGPT. When not to use AI.

Prompting for Code

Context → Task → Constraints → Verification. Four-part structure. Every time.

AI Hygiene

Always verify output. Read what Claude writes. Commit messages prove you understood the change.

Knowledge Contribution

The social contract: if you discover something useful, share it. This isn't optional — it's how the team compounds.

Hooks — the automation layer

Claude Code hooks let you auto-trigger actions at session events. Use them to:

  • → Auto-run /daybook when session ends
  • → Remind to push to git before closing
  • → Auto-load team CLAUDE.md context on session start

The Flywheel

When all four components work together, intelligence compounds

1 Intern uses CRISPY → structured work → quality output
2 Session log reflects on what was learned → goes into personal vault folder
3 Best insights promoted to Shared/ → benefit the whole cohort
4 Manager reviews git + vault weekly → catches issues early, reinforces patterns
5 Next cohort onboards to a vault full of patterns, dead ends, and institutional knowledge
6 Genuine framework gaps surface → manager reviews → ATLAS itself improves via PRs upstream
∞ Company intelligence compounds. The framework gets smarter with every cohort.

ATLAS Evolves With You

Every cohort improves the framework — automatically, with human review

How improvements surface

1 Daybook reflection surfaces a pattern — a guardrail that was missing, a skill that could be clearer
2 Claude evaluates: is this genuinely additive to ATLAS, or specific to this intern/project?
3 If genuine: "This looks like an ATLAS improvement — want me to raise a PR?"
4 Improvement applied to local ATLAS instance immediately. Manager reviews before anything merges upstream.

Role-aware behaviour

Intern

Morning nudge if yesterday's log is missing. Daybook surfaces improvement candidates. Works from their local ATLAS instance.

Manager

Session start shows cohort health. Reviews improvement PRs before they land upstream. Controls what propagates to the next cohort.

Role is set once during ./scripts/install.sh. No configuration drift.

Getting Started: Week 1

Manager

  • □ Obsidian vault in a Git repo
  • □ Folder structure per intern + Shared/ + Company/
  • □ Onboarding guide in Company/Onboarding/
  • □ PR + commit conventions documented
  • □ Weekly 30-min review scheduled

Each Intern

  • □ Clone vault, set up Obsidian
  • □ Read onboarding guide + CRISPY
  • □ Session log every day (no exceptions)
  • □ Clarify + Research phases before touching code
  • □ Push to git every day
Questions get written down — ask Claude first, escalate to manager with context. The writing-down part is not optional.

The Return on Investment

Cost

  • ~1 hour setup (vault + structure)
  • 5 min/day per intern (session log)
  • 30 min/week (manager review)
  • Week 1 AI foundations time

Return

  • Managers know what's happening without micromanaging
  • Mistakes caught at day 3, not week 6
  • Research compounds instead of evaporating
  • Each cohort starts smarter than the last
  • Interns produce work they can actually explain
  • Company builds genuine AI capability, not AI dependency
The cost is low. The asymmetry is enormous. The main risk is not implementing it and watching it happen by accident.
